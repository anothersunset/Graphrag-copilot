# v3.2 — Provenance Layer

v3.2 keeps the v3.1 7-node agentic graph intact and adds a *provenance
layer* on top: structured evidence, sentence-level citation, a new
sufficiency metric, and an adversarial harness.

## TL;DR upgrade map

| # | Surface | Before (v3.1) | After (v3.2) |
|---|---|---|---|
| 1 | Retrieval output | flat `list[RetrievalHit]` | `EvidencePack` with vector_chunks / graph_nodes / graph_paths / visited_nodes / rerank_trace |
| 2 | Auditor output | `cited_chunk_ids: list[str]` | + `claims: list[{text, evidence_ids, support}]` (sentence → evidence) |
| 3 | Eval metrics | 4 built-ins | + Provenance Sufficiency (no LLM, injectable entailer) |
| 4 | Robustness | none | adversarial harness with distractor injection (next commit) |
| 5 | Trace | hits + fused + cited | + `graph_path`, `visited_node_count`, `rewritten_query` per row |

## #1 EvidencePack contract

```python
class EvidencePack(BaseModel):
    vector_chunks: list[ChunkEvidence]
    graph_nodes:   list[GraphNode]
    graph_paths:   list[GraphPath]     # one per cited multi-hop traversal
    visited_nodes: list[GraphNode]     # union of every node the KG retriever touched
    rerank_trace:  list[RerankTraceRow]
```

The KG retriever now uses `MATCH p=(e:Entity)-[*1..max_depth]-()` with a
branch limit, so each hit carries a `GraphPath` with `nodes`, `rels` and
`depth`. `visited_nodes` is the *superset* — every node touched even if
its path didn't make the top-K. The adversarial harness uses it to
detect *"the misleading node was seen but correctly ignored"*.

## #2 Sentence-level claims

The DSPy auditor Signature emits a typed `claims` field:

```
claims: list[dict] = OutputField(desc=
    "list of {text, evidence_ids, support} dicts — one per sentence of"
    " draft_answer, where support is supported|partial|unsupported")
```

If DSPy fails or omits claims, `heuristic_claims()` rebuilds them with
token-overlap matching against the retrieval context. Sentence
segmentation is zh/en/jp punctuation-aware and preserves code fences +
display math.

State now exposes:

- `claims: list[Claim]` — per-sentence provenance
- `cited_chunk_ids: list[str]` — answer-level (unchanged, backward-compat)

AuditEntry.detail gains `claim_count` and `unsupported_claim_count` so
Langfuse and the audit drawer show per-sentence support.

## #3 Provenance Sufficiency metric

Definition: *if a reader saw only the cited chunks, could they
reproduce the answer?*

Approximation (pure Python, no LLM in the default path):

1. **Sentence-level recall** — each claim is sufficient when ≥1 of its
   `evidence_ids` is in `cited_chunk_ids` AND the cited chunk shares
   ≥ `min_overlap` significant tokens with the claim text.
2. **Answer-level coverage** — cited chunks' content must cover
   ≥ `coverage_floor` of the answer's significant tokens.

Final score: `recall_weight * sentence_recall + (1 − w) * coverage_bonus`.

The `entailer` hook accepts a BGE-Reranker-v2-m3 cross-encoder for
higher-fidelity offline evaluations — see `make_bge_entailer()`.

Targets (v3.2):

- `provenance_sufficiency_score ≥ 0.80`
- per-claim `unsupported_claim_count == 0` for `verdict=pass` runs

## #5 query_history end-to-end

The rewriter already wrote `query_rewrites: [new_query]` to state. v3.2
pipes that list through to `RetrievalTraceExporter.export(...,
query_history=...)` so every trace row knows the rewritten query that
produced it. The frontend can label each retrieval shell with
"rewrite #1: …".

## What's *not* in v3.2 (deliberately deferred)

- RBAC / risk-level on tools (out of scope — belongs to a separate
  Agent platform project, not this single-repo retrieval-augmented
  system).
- LLM-judge entailment as default (kept as an injectable hook, not the
  default path — we want CI to stay free).
